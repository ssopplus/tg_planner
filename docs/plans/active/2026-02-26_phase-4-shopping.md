## Контекст

Четвёртая фаза развития TG Planer (см. [docs/DOMAIN.md](../../DOMAIN.md)). Покупки выделяются в отдельный тип проекта с расширенным поведением: быстрый ввод через запятую, автокатегоризация по отделам магазина и запоминание частых покупок.

### Вопросы и ответы

**Q:** Как категоризировать товары — LLM или словарь?
**A:** Гибридный подход: словарь для частых товаров (бесплатно, быстро), LLM fallback для неизвестных.

**Q:** Нужны ли дедлайны и приоритеты для покупок?
**A:** Нет, покупки упрощены: только название, категория, количество, статус (куплено/нет).

## Решение

Проекты с типом SHOPPING получают специальное поведение: быстрый ввод списком через запятую, автокатегоризация через словарь + LLM fallback, отдельный UI в Mini App с группировкой по отделам. Система запоминает частые покупки и предлагает их при создании нового списка.

### Альтернативы

1. **Только LLM для категоризации** — медленно и дорого для каждого товара
2. **Только словарь** — не покроет нестандартные товары
3. **Покупки как обычные задачи** — теряется специфичное UX (быстрый ввод, категории, частые)

## Последствия

### Позитивные
- Список покупок — одна из самых частых задач, специальный UX значительно ускоряет работу
- Автокатегоризация экономит время в магазине (товары по отделам)
- Частые покупки — не нужно вспоминать что обычно покупаешь

### Риски
- Словарь нужно поддерживать и расширять
- LLM fallback добавляет latency для неизвестных товаров
- Нормализация названий может давать ложные совпадения ("молоко" vs "молоко топлёное")

### Технический долг
- Словарь захардкожен в коде (вынести в БД позже)
- Нет учёта магазинов (в каком магазине что купить)

## Тестирование

- "молоко, хлеб, яйца" → 3 отдельных товара в списке
- Товары автоматически попадают в правильные категории
- Неизвестный товар категоризируется через LLM fallback
- Частые покупки предлагаются после 3+ покупок
- "купить молоко" в боте → автоматически в проект "Покупки"
- В Mini App: товары сгруппированы по отделам
