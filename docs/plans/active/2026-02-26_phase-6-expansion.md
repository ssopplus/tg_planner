## Metadata

- **Date**: 2026-02-26

## Контекст

Шестая (финальная) фаза развития TG Planer (см. [docs/DOMAIN.md](../../DOMAIN.md)). Переход от личного инструмента к полноценной платформе: канбан-доска, multi-user с совместными списками, внешние интеграции (API, вебхуки, Google Calendar), аналитика продуктивности и мультиязычность.

### Вопросы и ответы

**Q:** В каком порядке реализовывать задачи фазы?
**A:** Рекомендуемый: канбан → API → Google Calendar → аналитика → multi-user → мультиязычность (multi-user и i18n — самые масштабные, в конце).

**Q:** Как реализовать реалтайм для совместных списков?
**A:** Polling с коротким интервалом для MVP, WebSocket/SSE позже при необходимости.

**Q:** Какой i18n фреймворк?
**A:** next-intl — нативная интеграция с Next.js App Router.

## Решение

Шесть независимых модулей: (1) канбан-доска с @dnd-kit между колонками, (2) multi-user через project_members с ролями OWNER/EDITOR/VIEWER и deep link приглашениями, (3) REST API v1 с API-ключами и исходящими вебхуками, (4) Google Calendar OAuth + двусторонняя синхронизация, (5) аналитика на основе user_activity_log с графиками в Mini App, (6) next-intl для русского и английского с автоопределением языка.

### Альтернативы

1. **WebSocket для реалтайма** — сложнее на Vercel serverless, polling достаточно для начала
2. **GraphQL вместо REST API** — overhead для текущих потребностей
3. **i18next вместо next-intl** — хуже интеграция с App Router

## Последствия

### Позитивные
- Канбан — визуальное управление проектами привычным способом
- Multi-user — совместные списки покупок и делегирование задач
- API — интеграция с любыми внешними сервисами
- Google Calendar — задачи видны в календаре
- Аналитика — мотивация через streaks и статистику
- Мультиязычность — расширение аудитории

### Риски
- Multi-user резко усложняет логику (права, уведомления, конфликты)
- Google Calendar API — сложная авторизация, quota limits
- Мультиязычность затрагивает весь код — высокий риск регрессий
- Vercel serverless ограничения для реалтайма multi-user

### Технический долг
- Polling вместо WebSocket для multi-user (временно)
- Нет конфликт-резолюции для одновременного редактирования
- OpenAPI документация может отставать от реализации
- Переводы нужно поддерживать при каждом изменении UI

## Тестирование

- Канбан: drag-n-drop между колонками меняет статус, порядок сохраняется
- API: CRUD операции через REST с API-ключом, rate limiting работает
- Вебхуки: при создании задачи приходит POST запрос на URL
- Google Calendar: задача с дедлайном появляется в календаре
- Аналитика: метрики корректны, streak считается правильно
- Multi-user: приглашение по ссылке, EDITOR может создавать задачи, VIEWER — нет
- Делегирование: назначенный получает уведомление
- Мультиязычность: переключение RU ↔ EN, все строки переведены
