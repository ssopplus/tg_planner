# Phase 5 — AI-фишки

> Источник требований: [docs/DOMAIN.md](../DOMAIN.md)

## Цель

Бот становится умным помощником: автоматически декомпозирует сложные задачи, замечает паттерны поведения пользователя, предлагает оптимальный порядок выполнения и поддерживает многоуровневые подзадачи. LLM-абстракция расширяется для поддержки разных провайдеров.

## Scope

- AI-декомпозиция задач
- Контекстные подсказки на основе паттернов
- Оптимальный порядок выполнения
- Расширение абстракции LLM-провайдера
- Многоуровневые подзадачи

## Пререквизиты

- Phase 1-4 завершены
- Подзадачи (один уровень) уже работают (Phase 3)
- LLM-абстракция с OpenAI провайдером настроена (Phase 1)
- Достаточно данных о поведении пользователя (история задач)

## Задачи

### 1. AI-декомпозиция задач

**Описание:** По запросу пользователя бот разбивает сложную задачу на подзадачи.

**Технические решения:**
- Команда в боте: "разбей на подзадачи" (ответом на задачу) или кнопка [Декомпозировать]
- В Mini App: кнопка "Разбить на подзадачи" на странице задачи
- LLM: используется мощная модель (GPT-4o / Claude Sonnet) — задача требует рассуждения
- Промпт включает: название задачи, описание, проект, контекст пользователя
- Результат: массив подзадач с названиями и опциональным порядком
- Подтверждение: показ предложенных подзадач + [Создать все] [Выбрать] [Отменить]

**Промпт-стратегия:**
- System: "Ты помощник по планированию задач. Разбей задачу на конкретные, выполнимые шаги."
- Ограничения: 3-10 подзадач, каждая — атомарное действие
- Контекст: существующие подзадачи (если есть), чтобы не дублировать

**Структура:**
```
src/lib/ai/
  prompts/
    decompose-task.ts     # Промпт для декомпозиции
  decomposer.ts           # Логика декомпозиции
```

**Зависимости:** Нет (первая задача фазы)

### 2. Контекстные подсказки (анализ паттернов)

**Описание:** Бот анализирует поведение пользователя и предлагает улучшения.

**Паттерны для обнаружения:**
- **Откладывание:** задача переносилась 3+ раз → "Может, разбить на части?"
- **Нереалистичное планирование:** регулярно не выполняет план дня → "Попробуй планировать меньше задач"
- **Забытые задачи:** задача без активности 14+ дней → "Актуально ли это?"
- **Продуктивные дни:** высокая активность → "Отличный день! Выполнено N задач"
- **Перегрузка:** слишком много задач с дедлайном на один день → "На завтра запланировано 15 задач, это реалистично?"

**Технические решения:**
- Анализ запускается при утреннем/вечернем дайджесте
- Отдельный сервис `PatternAnalyzer` с набором правил
- Правила — чистые функции: `(userHistory) => Suggestion | null`
- Хранение истории: таблица **user_activity_log** — user_id, action (task_created, task_completed, task_postponed, etc.), metadata (JSON), created_at
- Подсказки показываются в дайджесте как отдельная секция

**Структура:**
```
src/lib/ai/
  patterns/
    analyzer.ts           # Главный анализатор
    rules/
      postponing.ts       # Правило откладывания
      overplanning.ts     # Правило перепланирования
      forgotten.ts        # Правило забытых задач
      productivity.ts     # Правило продуктивности
      overload.ts         # Правило перегрузки
    types.ts              # Типы Suggestion, Rule
```

**Зависимости:** Нет (независима, но логически идёт после задачи 1)

### 3. Оптимальный порядок выполнения

**Описание:** Бот предлагает порядок, в котором лучше выполнять задачи дня.

**Технические решения:**
- Два подхода (гибрид):
  1. **Алгоритмический:** сортировка по дедлайну → приоритету → estimated effort
  2. **LLM-расширенный:** для сложных случаев LLM учитывает контекст ("утром лучше сложные задачи", "перед встречей не начинай большие")

- Факторы для сортировки:
  - Жёсткий дедлайн (время) — привязка к конкретному слоту
  - Приоритет (автоматический score из Phase 2)
  - Estimated complexity (если добавлено)
  - Зависимости между задачами (если задача A блокирует B)

- Интеграция: кнопка "Оптимизировать порядок" в "Мой день" (Mini App и бот)
- Результат: переупорядоченный список с кратким объяснением

**Зависимости:** Задача 1 (декомпозиция может влиять на порядок)

### 4. Расширение абстракции LLM-провайдера

**Описание:** Поддержка нескольких LLM-провайдеров с возможностью переключения.

**Технические решения:**
- Интерфейс `LLMProvider` (уже существует из Phase 1) расширяется:
  - `parse(text, context)` — парсинг задачи
  - `decompose(task, context)` — декомпозиция
  - `suggest(history)` — подсказки
  - `categorize(items)` — категоризация покупок
- Реализации:
  - `OpenAIProvider` (GPT-4o-mini, GPT-4o) — уже есть
  - `AnthropicProvider` (Haiku, Sonnet) — новый
  - Потенциально: `OllamaProvider` для локального запуска
- Конфигурация: какой провайдер для какой задачи
  - Парсинг: дешёвая модель (GPT-4o-mini / Haiku)
  - Декомпозиция: мощная модель (GPT-4o / Sonnet)
  - Подсказки: мощная модель

**Структура:**
```
src/lib/ai/
  provider.ts             # Обновлённый интерфейс
  providers/
    openai.ts             # OpenAI реализация
    anthropic.ts          # Anthropic реализация
  config.ts               # Маппинг задача → провайдер/модель
  factory.ts              # Фабрика провайдеров
```

**Настройки:**
- В `settings` или env: выбор провайдера по умолчанию
- Fallback: если основной провайдер недоступен → переключение на альтернативный

**Зависимости:** Нет (независимая инфраструктурная задача)

### 5. Многоуровневые подзадачи

**Описание:** Расширение подзадач с одного уровня до произвольной вложенности.

**Технические решения:**
- Изменение модели: subtasks получает поле `parent_subtask_id` (FK → subtasks, nullable)
- Рекурсивная структура: дерево подзадач
- Ограничение: максимум 3 уровня вложенности (для читаемости)
- Прогресс: рекурсивный расчёт (выполнены все подзадачи ветки → ветка выполнена)

**Изменения в UI (Mini App):**
- Древовидное отображение с indent
- Collapse/expand для веток
- Добавление подзадачи к подзадаче
- Прогресс-бар учитывает вложенность

**Изменения в боте:**
- Декомпозиция может создавать вложенные подзадачи
- Отображение: indent с символами `├──` и `└──`

**Зависимости:** Задача 1 (декомпозиция создаёт многоуровневые подзадачи)
